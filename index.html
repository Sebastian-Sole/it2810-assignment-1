A Mural Of Life

Information about the artwork

A Mural of Life, by painter Has Bulla, is an abstract depiction of life's greatest question: "What is the meaning of life?". The interactivity aspect of the digital artwork represents putting life into persepctive, and how life is bigger than just one person.

Use of Flexbox

I chose to use CSS-Flexbox since I have more experience with Flexbox vs CSS grid. I also believe that Flexbox is easier to deal with/better to use when there are few elements that need to be controlled. Another reason why I chose flexbox was because I wanted very basic styling, having everything centered on the page. This is easily achieveable with "justify-content: center" and "align-items: center" which center the child elements in the center both on the x and y axis. I use "flex-direction: column" to keep the column format that I wanted, which also made styling for responsive web design easier.

Use of HTML Canvas and SVG in HTML

I used SVG to create the shapes in the art, and jQuery to animate them on a click. I tried to use as many SVG elements as possible, i.e. circle, polygon and rect. I wanted to experiment with as many types of SVG elements, and I found polygon to be the trickiest. Most of the styling is done in the attributes of the SVG elements, but some of these change during the animation, especially the fill-opacity. All of the values which define the shapes size and location are hardcoded, something I realize is not good practice, but it made it easier for me to learn how SVG elements work, since I could mess around with specific values. Canvas, however, is all drawn with javascript. It took me a while to figure out that the width attribute in the canvas tag referred to the width of the canvas, and not the element. This produced some styling challenges, which I was able to fix after figuring out the problem. I decided to use object-oriented programming for the canvas elements, creating classes for circle, box and triangle. I did this to be able to store and access properties such as radius, fill-opacity, height and width for the squares, etc. This allowed me to easily change those values when animating. I realize now, however, that this produced a lot of duplicate, redundant, boilerplate code, and if I did this exercise again, I would plan out my workflow better. A lot of time was spent drawing one shape, animating it, trying to do the same for the next shape, and then having to change a lot of the code for the previous shape. Hardcoded values were used for the location and sizes of the shapes, which meant that I had to change a lot of the values after refactoring code, and working on resizing. Some elements, like the ones used for triangle rotation, are dynamically programmed. Canvas proved to be the most difficult to animate, since it works as a drawing canvas. If you want to move an element, you must erase what is there, and redraw it slightly diffrerent. It was also very frustrating that canvas only has a built in function to erase quadratic shapes, and not circular or triangular shapes. Having a rotating triangle without erasing the whole canvas was difficult, but proved to be achieveable with the help of requestFrameAnimation.

Use of jQuery

I used jQuery to access the HTML elements for both Canvas and SVG animation. I also used jQuery to animate the the SVG drawing when a user clicks on it, by using the ".animate()" function. jQuery reduced boilerplate code for accessing html elements, and although vanilla javascript was used for canvas animation, it's syntax was very easy to work with when animating svg elements. jQuery's library also makes it easier to handle events such as a click event by reducing boilerplate code, something which proved to be true when handling click events on the artwork. The jQuery library is used with the jQuery API. This was easier to work with, but it does mean that the animation does not work without internet connection. I don't see this as a problem, however.

Cross-Browser Testing

Cross-Browser testing was completed with Chrome and Safari (the two browsers that are installed on the computer). The testing was done by opening up the HTML file in both browsers, clicking the artwork to see if they would animate correctly, and by seeing that the responsive web design worked properly. In one instance, the rotation of the triangle was "choppy" in Safari, but smooth on Chrome. This was solved by adding a separate "transfom" style specifically for Safari, which was "-webkit-transform". Mobile testing was also done by using developer tools, and using the "Toggle device toolbar" functionality to resize the viewport to fit a smart phone device with a 480px screen.
